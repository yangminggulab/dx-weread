# 数据同步架构说明

## 三端概览

| 端 | 入口 | 数据存储 |
|---|---|---|
| 小程序 | 直接调用 Cloudflare Worker | 云端 KV（`app_data` / `diary_data`） |
| 网页版·云端模式 | `yangminggu.com/tasks`（Worker） | 云端 KV |
| 网页版·本地模式 | `localhost:8080`（server.py） | 本地 `data.json` / `diary.json` |

---

## 设计原则：本地只作备份

**云端是唯一权威**，小程序和云端网页版直接读写云端 KV。

本地 `server.py` 定位为**只读备份 + WeRead 同步桥**：
- 每 15 分钟从云端拉取最新数据存到本地
- 本地网页版的改动**只写本地，不推云端**
- 仅微信读书同步和每日重置会主动推云端

这个原则来自一个教训：曾经让本地每次保存都推云端，结果本地
`data.json` 还没来得及拉到最新状态（任务刚在小程序标为完成），
本地网页版触发保存 → 把旧的"未完成"数据推上去 → 覆盖了云端的
"已完成" → 小程序再打开任务又变回未完成。

## 数据流

### 任务（Tasks）

```
小程序
  └─ POST /api/tasks/add|update|delete ──→ 云端 KV
                                               │
网页云端版                                      │ 每15分钟单向拉取
  └─ POST /api/data ──────────────────→ 云端 KV │
                                               ↓
网页本地版                                 本地 data.json
  └─ POST /api/data ──→ 本地 data.json    （只读备份，不推回云端）
```

### 日记（Diary）

```
小程序
  └─ POST /api/diary ──────────────────→ 云端 KV
                                              │
网页云端版                                     │ 每15分钟单向拉取
  └─ POST /api/diary ────────────────→ 云端 KV │
                                              ↓
网页本地版                                本地 diary.json
  └─ POST /api/diary ──→ 本地 diary.json  （只读备份，不推回云端）
```

### 本地主动推云端的时机（例外情况）

| 触发条件 | 推送内容 | 原因 |
|---|---|---|
| 微信读书同步完成 | 任务 + 书单 | 只有本地能抓 WeRead 数据 |
| 凌晨5点每日重置 | 任务 + 书单 + 日记 | 清理完成任务后需同步 |
| 手动 `/api/diary/push` | 日记 | 手动触发 |
| 手动 `/api/sync/pull` | 从云端拉取（反向） | 手动触发 |

## 冲突优先级

### 任务

**云端为绝对权威**：
- 云端有、本地也有 → 比较 `updatedAt` / `createdAt`，**时间戳新的获胜**；无时间戳则云端获胜
- 云端有、本地没有 → 加入本地
- 云端没有、本地有 → **删除本地**（云端每日重置删掉的任务不会复活）

### 日记

**以内容更长的一方为准**（避免覆盖掉更完整的版本）：
- 今日内容：本地与云端各清洗一遍，**字数多的获胜**
- 归档条目：按日期去重，同一天**字数多的获胜**
- 归档后内容为空的条目会被丢弃

### 书单 / 笔记 / 动态

**本地为基础，云端补充**：
- 本地有、云端没有 → 保留本地
- 云端有、本地没有 → 加入本地
- 两端都有同一 ID → **时间戳新的获胜**；无时间戳则本地获胜

## 归档顺序

### 日记归档

- **触发条件**：今日日记的 `date` 字段与当前日期不符（跨天了）
- **排列顺序**：按 `date` 字符串升序（最早的在最前），归档列表永不删除
- **归档后**：`today` 重置为当天空内容
- **分界线**：凌晨5点前仍算昨天（本地和云端一致）

### 每日重置执行顺序（本地）

1. 从云端拉取最新数据并合并
2. 归档日记
3. 推送日记到云端
4. 清除已完成任务
5. 推送任务到云端

---

## 每日重置

两处独立运行，逻辑相同：

| | 云端 | 本地 |
|---|---|---|
| 时间 | Cron `0 21 * * *` UTC（北京时间5点） | 每天5点本地时区 |
| 操作 | 归档日记 + 清除已完成任务 | 归档日记 + 清除已完成任务 + 推送云端 |

## 已知问题

### ⚠️ 日记归档时间依赖本地时区

`server.py` 的 `effective_diary_date()` 使用 `datetime.now()`（无时区），若 Mac 时区不是 CST（UTC+8）会导致归档日期与云端不一致。目前 Mac 在 CST，暂无影响。
